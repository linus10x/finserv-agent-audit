## Context

Any SEC-registrant adopter of this repository is operating under Section 404 of the Sarbanes-Oxley Act of 2002 (15 U.S.C. § 7262). Management must annually assess — and the external auditor must attest to — the effectiveness of internal control over financial reporting (ICFR). For an AI-mediated workflow that touches a financial-reporting flow (revenue recognition, fair-value mark, customer-credit reserve, trading P&L, AML transaction record), the agent and its supporting infrastructure are in the ICFR scope. The PCAOB Auditing Standard 2201 framework that auditors apply expects IT general controls (ITGC) over the systems that produce the financial data.

The PCAOB and SEC have, in practice, settled on four ITGC categories (the same four that COBIT 2019 and the AICPA's Trust Services Criteria use): **access controls, change management, computer operations, and program development**. A repository that ships an AI-governance pattern catalog without a named mapping to these four categories forces every SEC-registrant adopter to do the mapping privately — and audit risk is the asymmetric cost of getting it wrong.

The 2026-05-28 internal audit named this gap (Finding D4.2, Critical). This ADR closes it.

## Decision

Adopt this ADR as the authoritative mapping of every pattern in this repository to the four SOX 404 ITGC categories. The overlay is documentary, not executable: it links pattern → ITGC category → control objective → audit-evidence artifact the pattern already produces. Adopters use the mapping when populating their internal-control matrix (ICM) and when preparing PBC (prepared-by-client) lists for the external auditor.

### ITGC overlay

| ITGC Category | Control Objective (PCAOB / COBIT framing) | Pattern in this repo | Audit-evidence artifact |
|---|---|---|---|
| **Access Controls** | Only authorized individuals can access systems and data; segregation of duties enforced | Sovereign Veto (ADR-0002) — operator-clearance gate with named bypass owner; DEFCON-1 operator-only mode | Hash-chained audit entries with `bypass_owner` field; DEFCON transition logs with named actor |
| **Change Management** | Changes to applications and infrastructure are authorized, tested, and approved before production | Shadow Mode (ADR-0006) — pre-promotion divergence testing; Autonomy Ladder (ADR-0004) A0→A4 promotion gates; MI Proxy (ADR-0015) verifier integrity on the audit pipeline itself | Shadow-mode divergence reports; A-level promotion ADRs; MI Proxy attestation records |
| **Computer Operations** | Production jobs run as scheduled; failures detected and resolved; backups and recovery tested | DEFCON state machine (ADR-0001) — system-wide operational posture; hash-chained audit ledger (ADR-0003) with WORM persistence (ADR-0013) | DEFCON transition timeline; ledger retention reports; WORM-store retention attestation |
| **Program Development** | New systems and major changes follow a documented SDLC with testing and approval | Autonomy Ladder (ADR-0004) governance gates; Vendor Score Gate (ADR-0016) for third-party model onboarding; Persistence/Witness/Timestamp pattern (ADR-0014) for the audit pipeline itself | A-level promotion records; vendor onboarding decision logs; witness-anchor receipts on chain-head digests |

### Per-pattern detail

Every ADR in this repository carries a `## Regulatory Mapping` section. Adopters operating under SOX 404 should treat the mapping above as a starting point and extend it with their firm's control identifiers (e.g., `ITGC-AC-007`, `ITGC-CM-014`) in their internal-control matrix. The pattern code does not change; the firm's ICM cross-references the pattern.

### What this ADR does NOT do

- Prescribe a specific test-of-design or test-of-operating-effectiveness procedure. Those belong to the firm's internal-audit and external-audit teams.
- Substitute for SOC 1 Type 2 reports on third-party providers in the ICFR scope. Third-party SOC 1 reports remain a separate procurement requirement; this overlay does not displace them.
- Map to COSO 2013 entity-level controls. The COSO mapping is broader than ITGC and warrants a separate ADR if buyer-side demand surfaces.

## Alternatives Considered

1. **Bury the SOX mapping in each per-pattern ADR.** Rejected — produces duplication and drift. A SOX 404 auditor wants one document, not seventeen, when populating the ITGC scoping memo.
2. **Defer SOX 404 mapping until a buyer requests it.** Rejected — the buyer is the SEC-registrant adopter, and the registrant's external auditor will request the artifact within the first interim review. Deferred-on-request is too late.
3. **Use the AICPA Trust Services Criteria (TSC) instead of the four ITGC categories.** Rejected — TSC is the SOC 2 vocabulary; the SOX 404 ITGC vocabulary is what registered-firm auditors apply. A separate SOC 2 mapping ADR is appropriate but does not substitute for this one.
4. **Map to COBIT 2019 process areas directly.** Rejected — COBIT framing is broader than what a PCAOB AS 2201 walkthrough actually tests. The four-category mapping is the proven framing.

## Consequences

**Positive.** A SEC-registrant adopter populates an internal-control matrix entry in minutes, not weeks. The external auditor's PBC list maps cleanly to artifacts the patterns already produce — no special extraction step is required at year-end. The overlay is a procurement-side artifact a CIO or Chief Compliance Officer can hand to internal audit without engineering involvement.

**Negative.** PCAOB inspection findings have, in recent years, intensified expectations around ITGC documentation precision and around the operating-effectiveness evidence trail. The pattern code produces the artifacts; the firm's audit-evidence workflow must capture and retain them per the firm's retention schedule (linked to ADR on retention, when published). A pattern that produces the right artifact but a firm that fails to retain it is still a deficiency.

**Operational.** The overlay assumes the firm has a populated internal-control matrix and a defined audit-evidence workflow. Firms early in their SOX 404 maturity (e.g., recent IPOs, recent acquisitions of a regulated subsidiary) need internal-audit involvement before adoption.

## Pre-mortem

What fails:

1. **Auditor disagrees with the four-category mapping.** Mitigation — the mapping is a starting point; the firm's external auditor's framing controls. The ADR documents the rationale and can be updated by superseding ADR.
2. **A pattern produces the right artifact, but the artifact is stored in a non-WORM substrate.** Mitigation — ADR-0013 (SEC 17a-4 WORM) addresses the retention substrate; the cross-reference points adopters at that ADR.
3. **A new pattern lands without an ITGC-category row.** Mitigation — the contributing guide will require an ITGC row on any ADR that materially changes the audit-evidence shape. (Tracked for `CONTRIBUTING.md` update in Tranche 3.)
4. **External auditor cites this ADR as audit-firm-approved guidance.** Mitigation — the disclaimer at the top is explicit; the ADR is a reference pattern, not auditor-approved methodology.

## Reversibility

High. The overlay is documentary. Withdrawing it removes the table; runtime patterns are unaffected. The cost of reversal is loss of buyer-side artifact, not loss of safety.
